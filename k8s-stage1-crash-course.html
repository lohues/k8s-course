// 01

Linux Basics

Kubernetes runs on Linux. Every node in your cluster is a Linux machine. Understanding the filesystem, processes, permissions, and basic commands is non-negotiable.

// The Filesystem

Linux has one unified directory tree starting at /. Everything — devices, processes, configs, logs — is a file somewhere in this tree.

/ root of the entire filesystem
├── bin/ essential binaries (ls, cp, bash)
├── etc/ system-wide config files
├── home/ user home directories
├── proc/ virtual FS — running process info (used by K8s!)
├── sys/ virtual FS — kernel/hardware info
├── tmp/ temporary files, cleared on reboot
├── usr/ user-installed programs and libraries
├── var/ logs, caches, runtime data
│ └── log/ system and app logs (syslog, auth.log)
└── mnt/ mount points for volumes (important in K8s!)

// File Permissions

Every file has permissions for three actors: the owner, the group, and others. Kubernetes uses this heavily for mounted secrets and config files.

bash · understanding permissions
# Output from: ls -la
-rwxr-xr--  1 alice devs  4096  Feb 25 10:00  script.sh
 ↑↑↑↑↑↑↑↑↑
 │└──┬──┘└──┬──┘└──┬──┘
 │ owner  group  others
 └─ file type: - (file), d (dir), l (symlink)

# r=read(4) w=write(2) x=execute(1)
# rwxr-xr-- means: owner=rwx(7) group=r-x(5) others=r--(4) → 754

# Change permissions (chmod)
chmod 755 script.sh       # numeric: rwxr-xr-x
chmod +x script.sh        # add execute for all
chmod 600 secret.key      # owner read/write only (for SSH keys)

# Change owner (chown)
chown alice:devs file.txt  # change owner and group
chown -R alice ./dir/     # recursively

// 02

Docker & Containers

Kubernetes orchestrates containers. Docker is the most common tool for building and running them. Before Kubernetes makes sense, containers must be completely natural to you.

// Containers vs Virtual Machines

Virtual Machine
Includes full OS (kernel + userland)
Gigabytes in size
Boots in 30–60 seconds
Strong hardware-level isolation
Managed by hypervisor (VMware, KVM)
Container
Shares the host kernel, packages only app + deps
Megabytes in size
Starts in milliseconds
Process-level isolation via namespaces + cgroups
Managed by container runtime (containerd)

// Writing a Dockerfile

A Dockerfile is a recipe for building a container image. Each instruction creates a new read-only layer — understanding layers is key to building efficient images.

dockerfile · node.js app example
# Base image — use official, minimal images
FROM node:20-alpine

# Set working directory inside the container
WORKDIR /app

# Copy package files first (layer caching optimization!)
COPY package*.json ./

# Install dependencies
RUN npm ci --only=production

# Copy the rest of the app
COPY . .

# Don't run as root inside containers
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Document the port (doesn't actually expose it)
EXPOSE 3000

# Default command to run the app
CMD ["node", "server.js"]
⚠️ Layer Caching

Docker caches each layer. Put things that change least often at the top (base image, dependencies) and things that change most often at the bottom (source code). This dramatically speeds up rebuilds.

// Docker Compose

Docker Compose runs multi-container apps locally. Think of it as a mini-Kubernetes for your laptop. Understanding Compose helps you grasp K8s concepts like Services and Deployments.

yaml · docker-compose.yml
version: '3.9'
services:
  web:
    build: .                # build from Dockerfile
    ports: ["3000:3000"]
    environment:
      DB_HOST: postgres    # service name = hostname in Docker network
      DB_PORT: "5432"
    depends_on: [postgres]

  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:   # named volume (persists between container restarts)
bash · docker compose commands
docker compose up -d      # start all services in background
docker compose down        # stop and remove containers
docker compose down -v     # also delete volumes
docker compose logs -f web # follow logs for 'web' service
docker compose ps          # list running services
docker compose exec web sh # shell into 'web' container
🔗 Compose → Kubernetes

When you move from Docker Compose to Kubernetes: services become K8s Services, image + ports become Pod specs in Deployments, volumes become PersistentVolumeClaims, and environment becomes ConfigMaps and Secrets.

// 03

Networking Fundamentals

Kubernetes networking is complex. Before tackling Services, Ingress, and NetworkPolicies, you need solid grounding in the basics that underpin all of it.

// IP Addresses & Subnets

Every machine (and container, and Pod) has an IP address. A subnet is a range of IPs in the same network. CIDR notation describes these ranges.

bash · IP and network basics
# CIDR notation: IP/prefix-length
192.168.1.0/24  → 256 addresses (192.168.1.0 – 192.168.1.255)
10.0.0.0/16     → 65536 addresses (common for K8s pod CIDR)
172.16.0.0/12   → 1M+ addresses

# K8s commonly uses:
# Pod CIDR:     10.244.0.0/16 (Flannel) or 192.168.0.0/16 (Calico)
# Service CIDR: 10.96.0.0/12

# View your machine's IP addresses
ip addr show
ifconfig       # older systems

# View routing table
ip route show
route -n

// DNS — Domain Name System

DNS translates names like my-service.default.svc.cluster.local into IP addresses. Kubernetes runs its own DNS server (CoreDNS) that makes service discovery work automatically.

bash · DNS debugging tools
# Resolve a hostname
nslookup google.com
dig google.com
host google.com

# Trace the full DNS resolution path
dig +trace kubernetes.io

# Check what DNS server you're using
cat /etc/resolv.conf
nameserver 10.96.0.10    ← this is CoreDNS inside a K8s Pod!
search default.svc.cluster.local svc.cluster.local cluster.local

# Test DNS from inside a K8s pod
kubectl run debug --image=busybox --rm -it -- nslookup kubernetes.default

// Networking Debugging Toolkit

bash · essential network commands
# Test connectivity
ping 8.8.8.8                # is this host reachable?
curl -I https://google.com  # HTTP response headers

# Trace route to destination

# Inspect network interfaces

# View iptables rules (what K8s kube-proxy manages)
iptables -L -n -v
💡 K8s Networking Rules

Kubernetes enforces 3 rules: Pod-to-Pod communication needs no NAT. Node-to-Pod communication needs no NAT. The IP a Pod sees itself as is the same IP others use to reach it. Everything is flat — no masquerading.

// 04

YAML

Every Kubernetes resource is defined in YAML. A single indentation error can break a deployment. You must be comfortable reading and writing YAML before touching kubectl apply.

// YAML vs JSON

YAML is a superset of JSON — valid JSON is also valid YAML. YAML uses indentation instead of braces and is designed for human readability. Kubernetes accepts both, but YAML is standard.

JSON
{
  "name": "nginx",
  "version": "1.25",
  "replicas": 3,
  "enabled": true,
  "ports": [80, 443],
  "labels": {
    "app": "web",
    "env": "prod"
  }
}
YAML (equivalent)
name: nginx
version: "1.25"
replicas: 3
enabled: true
ports: [80, 443]
labels:
  app: web
  env: prod

// YAML Syntax Rules

yaml · complete syntax reference
# 1. Comments start with #
# 2. ALWAYS use spaces — NEVER tabs
# 3. Indentation defines structure (2 spaces per level is standard)

---   # document separator (optional but good practice)

# Scalars (strings, numbers, booleans)
name: my-app             # bare string
version: "1.0.0"         # quoted string (use when has special chars)
replicas: 3              # integer
cpu_limit: 0.5           # float
enabled: true            # boolean (true/false, yes/no, on/off)
nothing: null            # null value (~  also works)

# Nested mapping (object)
resources:
  requests:
    cpu: "250m"
    memory: "256Mi"

# Sequence (list) — block style
ports:
  - 80
  - 443
  - 8080

# Sequence — flow style (inline)
ports: [80, 443, 8080]

# List of objects
containers:
  - name: web
    image: nginx:1.25
    ports:
      - containerPort: 80
  - name: sidecar
    image: fluentd:latest

# Multiline strings
config_file: |          # | preserves newlines (literal block)
  server {
    listen 80;
    location / { proxy_pass http://backend; }
  }

description: >          # > folds newlines into spaces (folded block)
  This is a very long
  description that will
  become one line.

// Common YAML Gotchas

yaml · pitfalls to avoid
# ❌ WRONG: using tabs
name: app
	image: nginx   ← TAB here causes parse error!

# ✅ RIGHT: use spaces
name: app
  image: nginx

# ❌ WRONG: inconsistent indentation
spec:
  containers:
  - name: app
      image: nginx   ← wrong indent level!

# ❌ Strings that look like other types need quoting
version: 1.10    ← parsed as float 1.1, not string "1.10"!
version: "1.10"  # ✅ correct

value: yes       ← parsed as boolean true, not string "yes"!
value: "yes"    # ✅ correct

# Validate YAML before applying
kubectl apply --dry-run=client -f deployment.yaml
python3 -c "import yaml,sys; yaml.safe_load(open('file.yaml'))"

// YAML Anchors & Aliases (DRY YAML)

YAML supports anchors (&) and aliases (*) to reuse content and avoid repetition — useful in CI/CD configs.

yaml · anchors & aliases
# Define an anchor with &
defaults: &defaults
  restartPolicy: Always
  imagePullPolicy: IfNotPresent

# Reuse it with * (alias) and << (merge)
production:
  <<: *defaults       # merges all keys from defaults
  replicas: 5

staging:
  <<: *defaults
  replicas: 1
🛠️ Helpful Tools

Use yamllint to catch syntax errors. The YAML Language Server extension for VS Code gives real-time validation and Kubernetes schema completions. Install it — it will save you hours of debugging.

// Shell Scripting Basics

You'll write scripts to automate kubectl commands, set up environments, and run batch operations. These fundamentals are enough to get started.

bash · shell scripting essentials
#!/bin/bash              # shebang line — always first
set -euo pipefail       # exit on error, undefined vars, pipe failures

# Variables
NAMESPACE="production"
APP="my-app"
REPLICAS=5

# String interpolation
echo "Scaling $APP to $REPLICAS replicas in $NAMESPACE"

# Conditionals
if kubectl get namespace "$NAMESPACE" &>/dev/null; then
  echo "Namespace exists"
else
  kubectl create namespace "$NAMESPACE"
fi

# Loops
for pod in $(kubectl get pods -n "$NAMESPACE" -o name); do
  echo "Checking: $pod"
  kubectl logs "$pod" -n "$NAMESPACE" --tail=5
done

# Capture command output
CURRENT_CONTEXT=$(kubectl config current-context)
echo "Operating on cluster: $CURRENT_CONTEXT"

# Functions
wait_for_rollout() {
  local deployment=$1
  local namespace=${2:-default}
  echo "Waiting for $deployment..."
  kubectl rollout status deploy/"$deployment" -n "$namespace"
}

wait_for_rollout "my-app" "production"
