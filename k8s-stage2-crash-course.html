// 01

Why Kubernetes?

Before containers became ubiquitous, applications were deployed directly on servers. Teams managed dependencies, configs, and scaling manually — a nightmare at scale.

Docker solved packaging. But running hundreds of containers across many machines, with load balancing, self-healing, and zero-downtime deploys? That's where Kubernetes comes in.

// The Problems K8s Solves

⚖️
Load Balancing
Distribute traffic across healthy replicas automatically. No nginx config files managed by hand.
🔄
Self-Healing
Crashed container? K8s detects and restarts it. Node dies? Pods get rescheduled elsewhere.
📈
Scaling
Scale from 2 to 200 replicas with one command (or automatically based on CPU/memory).
🚀
Rolling Deploys
Deploy new versions with zero downtime. Instant rollback if something breaks.
📌 Key Insight

Kubernetes is a container orchestrator — it manages the what, where, and when of running containers, so you don't have to.

// Control Plane Components

API Server — the front door. Every request (from kubectl, dashboards, CI/CD) goes through it. It validates and stores state in etcd.

Scheduler — watches for unscheduled Pods and assigns them to a suitable Node based on resource availability, constraints, and policies.

Controller Manager — runs control loops that constantly compare the desired state (what you declared) vs actual state (what's running) and corrects the drift.

etcd — a distributed key-value store. This is the cluster's single source of truth. Back this up!

// Worker Node Components

kubelet — an agent on every node. It talks to the API server, pulls container images, and ensures Pods are running healthy.

kube-proxy — handles networking rules on each node. Manages Service IP routing using iptables or IPVS.

Container Runtime — the engine that actually runs containers (containerd, CRI-O, or Docker).

💡 Mental Model

Think of the Control Plane as a factory manager and Worker Nodes as factory floor workers. The manager assigns jobs; the workers execute them.

// 03

Core Kubernetes Objects

Everything in Kubernetes is an object — a record of intent. You declare what you want, K8s makes it happen and keeps it that way.

// Pod

The smallest deployable unit. A Pod wraps one or more containers that share network and storage. In practice, most Pods contain a single container.

yaml · pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  containers:
  - name: web
    image: nginx:1.25
    ports:
    - containerPort: 80
⚠️ Important

Never deploy standalone Pods in production. Use a Deployment instead — bare Pods are not rescheduled if a Node fails.

// ReplicaSet

Ensures a specified number of Pod replicas are running at all times. If a Pod dies, ReplicaSet creates a new one. You rarely use ReplicaSets directly — Deployments manage them for you.

// Deployment

The standard way to run stateless apps. Wraps a ReplicaSet and adds rolling updates and rollback capabilities.

yaml · deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3             # run 3 pods
  selector:
    matchLabels:
      app: my-app
  template:               # pod template
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # 1 extra pod during update
      maxUnavailable: 0   # never less than desired

// Service

Pods are ephemeral — they get new IPs when restarted. A Service provides a stable network endpoint (DNS name + IP) that load-balances traffic to matching Pods.

ClusterIP
Default. Exposes Service on an internal IP. Only reachable within the cluster.
internaldefault
NodePort
Opens a port (30000–32767) on every Node. Useful for dev/testing. Not for production.
externaldev
LoadBalancer
Provisions a cloud load balancer (AWS ELB, GCP LB). The standard way to expose apps publicly.
externalcloud
yaml · service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  selector:
    app: my-app      # routes to pods with this label
  ports:
  - port: 80          # service port
    targetPort: 80   # container port
  type: ClusterIP

// Namespace

Namespaces provide virtual clusters within a physical cluster. Great for separating teams, environments (dev/staging), or applications.

bash · namespace commands
# Create a namespace
kubectl create namespace staging

# Deploy into a specific namespace
kubectl apply -f deployment.yaml -n staging

# List all namespaces
kubectl get namespaces

# Default namespaces in every cluster:
default         → your resources go here if unspecified
kube-system     → K8s internal components
kube-public     → publicly readable resources
kube-node-lease → node heartbeat objects

alias k=kubectl

kubectl get pod my-pod -o json    # JSON output
// 05

ConfigMaps & Secrets

Hardcoding config and credentials into container images is bad practice. Kubernetes provides first-class objects to inject configuration at runtime.

// ConfigMap

Stores non-sensitive configuration data as key-value pairs. Inject them as environment variables or mount as files.

yaml · configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  LOG_LEVEL: info
  MAX_CONNECTIONS: "100"
  app.properties: |    # can store whole files too
    server.port=8080
    cache.ttl=300
yaml · using configmap in a pod
spec:
  containers:
  - name: app
    image: my-app:1.0
    envFrom:
    - configMapRef:       # inject ALL keys as env vars
        name: app-config
    env:
    - name: LOG_LEVEL       # inject specific key
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: LOG_LEVEL

// Secret

Like ConfigMaps but for sensitive data (passwords, tokens, TLS certs). Values are base64-encoded at rest. For real security, use external secret managers (Vault, AWS Secrets Manager).

bash + yaml · creating secrets
# Create from literal (kubectl handles base64 encoding)
kubectl create secret generic db-creds \
  --from-literal=DB_USER=admin \
  --from-literal=DB_PASS=supersecret

# Or define in YAML (you encode manually with: echo -n 'value' | base64)
apiVersion: v1
kind: Secret
metadata:
  name: db-creds
type: Opaque
data:
  DB_USER: YWRtaW4=          # base64 of "admin"
  DB_PASS: c3VwZXJzZWNyZXQ=  # base64 of "supersecret"
⚠️ Security Note

Base64 is NOT encryption. Secrets are only as secure as your cluster's RBAC. In production, always use encrypted etcd and external secret management.

// 06

Labels, Selectors & Annotations

Labels are how Kubernetes objects find and connect to each other. Mastering labels is essential for understanding Services, Deployments, and more.

// Labels

Key-value pairs attached to any object. Used to organize, select, and filter resources.

yaml · common label patterns
metadata:
  labels:
    app: my-app           # app name
    version: v2.1.0       # semantic version
    environment: staging  # deployment env
    tier: backend         # app layer
    team: payments        # owning team

// Selectors

Used by Services, Deployments, and ReplicaSets to find target Pods. This is the glue that connects objects together.

bash · selecting with kubectl
# Get pods by label
kubectl get pods -l app=my-app
kubectl get pods -l environment=staging,tier=backend

# Set-based selectors
kubectl get pods -l 'environment in (staging, production)'
kubectl get pods -l 'version notin (v1.0, v1.1)'

# Add a label to existing resource
kubectl label pod my-pod tier=backend

# Remove a label (note the minus)
kubectl label pod my-pod tier-

// Annotations

Like labels, but for non-identifying metadata. Not used for selection. Used for tooling, documentation, and config (e.g., Ingress config, monitoring scrape paths).

yaml · annotations example
metadata:
  annotations:
    kubernetes.io/change-cause: "Bump nginx to 1.25 for CVE fix"
    prometheus.io/scrape: "true"
    prometheus.io/port: "9090"
    owner: "platform-team@company.com"

Hands-On Lab

Theory is useless without practice. Follow these steps to deploy a real app on a local cluster.

// Setup

bash · install minikube
# macOS
brew install minikube kubectl

# Linux
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Start your cluster
minikube start

# Verify it's running
kubectl cluster-info
Kubernetes control plane is running at https://127.0.0.1:8443

// Lab Steps

1

Create a Deployment

Deploy a simple nginx app with 3 replicas.

bash
kubectl create deployment webapp --image=nginx:1.25 --replicas=3
kubectl get deployments
kubectl get pods
2

Expose it with a Service

Create a NodePort service to access it from your browser.

bash
kubectl expose deployment webapp --port=80 --type=NodePort
minikube service webapp --url   # opens in browser
3

Simulate a Rolling Update

Update the image and watch K8s roll it out without downtime.

bash
kubectl set image deployment/webapp nginx=nginx:alpine
kubectl rollout status deployment/webapp
Waiting for deployment "webapp" rollout to finish...
deployment "webapp" successfully rolled out
4

Test Self-Healing

Delete a pod and watch it get recreated automatically.

bash
# In terminal 1: watch pods
kubectl get pods -w

# In terminal 2: delete a pod
kubectl delete pod <pod-name>

# Watch a new one spin up instantly in terminal 1!
5

Add a ConfigMap

Create a config and mount it into the deployment.

bash
kubectl create configmap app-cfg --from-literal=ENV=production
kubectl set env deployment/webapp --from=configmap/app-cfg
kubectl exec -it <pod> -- env | grep ENV
ENV=production
