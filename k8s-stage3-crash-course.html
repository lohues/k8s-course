// 01

Scheduling

The Kubernetes Scheduler assigns Pods to Nodes. By default it picks the most suitable node automatically — but you can influence or override this with several mechanisms.

// Node Selectors

The simplest way to constrain a Pod to run on Nodes with specific labels. Good for basic requirements like "run only on SSD nodes" or "run on GPU nodes".

bash · label a node first
# Add a label to a node
kubectl label node worker-1 disktype=ssd
kubectl label node worker-2 gpu=true
yaml · pod with nodeSelector
spec:
  nodeSelector:
    disktype: ssd    # pod ONLY runs on ssd-labeled nodes
  containers:
  - name: app
    image: my-app:1.0

// Node Affinity

More powerful than nodeSelector. Supports complex expressions and two modes: required (hard) and preferred (soft).

yaml · node affinity
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:   # HARD rule
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In
            values: [ssd, nvme]
      preferredDuringSchedulingIgnoredDuringExecution:  # SOFT rule
      - weight: 80
        preference:
          matchExpressions:
          - key: zone
            operator: In
            values: [us-east-1a]

// Pod Affinity & Anti-Affinity

Co-locate Pods together (affinity) or spread them apart (anti-affinity) based on what's already running on a node.

yaml · pod anti-affinity (spread replicas across nodes)
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: my-app     # don't place 2 replicas of same app on same node
        topologyKey: kubernetes.io/hostname

// Taints & Tolerations

The inverse of affinity. Taints are applied to Nodes to repel Pods. Tolerations are applied to Pods to allow them onto tainted nodes. Used to dedicate nodes for specific workloads.

Taint & Toleration Flow
Node: gpu=true:NoSchedule
←
Pod with toleration
✓ Scheduled
Node: gpu=true:NoSchedule
←
Pod without toleration
✗ Rejected
bash + yaml · taints & tolerations
# Apply a taint to a node
kubectl taint node worker-3 gpu=true:NoSchedule

# Taint effects:
NoSchedule        → new pods without toleration won't be placed here
PreferNoSchedule  → soft version, avoid if possible
NoExecute         → evicts existing pods too (+ blocks new ones)

# Remove a taint (note the minus at the end)
kubectl taint node worker-3 gpu=true:NoSchedule-

# Add toleration in Pod spec to allow scheduling on tainted node
spec:
  tolerations:
  - key: "gpu"
    operator: "Equal"
    value: "true"
    effect: "NoSchedule"
💡 Real World Use

Control Plane nodes are tainted with node-role.kubernetes.io/control-plane:NoSchedule by default. That's why your workloads only run on Worker Nodes.

Storage: PV, PVC & StorageClass

Container storage is ephemeral by default — data disappears when a Pod restarts. Kubernetes has a layered storage model for persistent data.

Kubernetes Storage Model
StorageClass
→ provisions →
PersistentVolume (PV)
→ binds to →
PersistentVolumeClaim (PVC)
→ mounted in →
Pod

// PersistentVolume (PV)

A piece of storage in the cluster provisioned by an admin (or dynamically by a StorageClass). It's a cluster-level resource — not tied to any namespace.

yaml · pv.yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce       # RWO: one node at a time
  persistentVolumeReclaimPolicy: Retain  # keep data after PVC deleted
  storageClassName: standard
  hostPath:                          # for dev/testing only
    path: /mnt/data
Access ModeShortDescription
ReadWriteOnceRWORead/write by a single node. Most common for databases.
ReadOnlyManyROXRead-only by many nodes simultaneously.
ReadWriteManyRWXRead/write by many nodes. Requires NFS or distributed storage.
ReadWriteOncePodRWOPRead/write by a single Pod only (K8s 1.22+).

// PersistentVolumeClaim (PVC)

A request for storage by a user/application. K8s finds a matching PV and binds them. PVCs are namespace-scoped.

yaml · pvc.yaml + using in a pod
# 1. Create the claim
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard
---
# 2. Use the claim in a Pod
spec:
  volumes:
  - name: my-storage
    persistentVolumeClaim:
      claimName: my-pvc
  containers:
  - name: app
    image: postgres:15
    volumeMounts:
    - mountPath: /var/lib/postgresql/data
      name: my-storage

// StorageClass

Enables dynamic provisioning — automatically creates PVs on demand when a PVC requests storage. Cloud providers supply their own StorageClasses (AWS gp3, GCP pd-ssd, etc.).

yaml · storageclass.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com  # AWS EBS CSI driver
parameters:
  type: gp3
  iops: "3000"
  throughput: "125"
reclaimPolicy: Delete       # delete PV when PVC is deleted
allowVolumeExpansion: true  # allow resizing
volumeBindingMode: WaitForFirstConsumer  # provision when Pod is scheduled
⚠️ Reclaim Policies

Retain — manual cleanup required after PVC deletion. Safe for production.  |  Delete — PV and underlying storage deleted with PVC. Convenient for dev.  |  Recycle — deprecated, don't use.

// 03

Networking & Ingress

Kubernetes networking has a key rule: every Pod gets its own IP and can talk to any other Pod without NAT. But Services and Ingress are how you expose things properly.

// In-Cluster DNS

CoreDNS runs in every cluster and gives Services automatic DNS names. You can reach any Service by name without hardcoding IPs.

bash · DNS resolution patterns
# Full DNS format: {service}.{namespace}.svc.cluster.local

# Same namespace — just use service name
curl http://my-api

# Cross-namespace — use full name
curl http://my-api.payments.svc.cluster.local

# Test DNS from inside a pod
kubectl exec -it my-pod -- nslookup my-api
kubectl exec -it my-pod -- curl http://my-api/health

// Ingress

An Ingress exposes HTTP/HTTPS routes from outside the cluster to Services inside. Think of it as a smart reverse proxy — one external IP, many internal services.

Ingress Traffic Flow
Internet
→
LoadBalancer Service
→
Ingress Controller
(nginx / traefik)
→
Ingress Rules
/api/* → api-service:8080
/web/* → web-service:3000
admin.domain.com → admin-svc:9090
bash · install nginx ingress controller (minikube)
minikube addons enable ingress

# Or on a real cluster:
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm install ingress-nginx ingress-nginx/ingress-nginx
yaml · ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port: { number: 8080 }
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-service
            port: { number: 3000 }
  tls:
  - hosts: [myapp.example.com]
    secretName: tls-secret  # TLS cert stored as a K8s Secret

// NetworkPolicies

By default, all Pods can talk to all other Pods. NetworkPolicies act as firewall rules — restrict which Pods can communicate with each other.

yaml · networkpolicy — deny all, then allow specific
# Step 1: Deny all ingress to 'backend' pods
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes: [Ingress]   # no ingress rules = deny all
---
# Step 2: Allow only 'frontend' pods to reach 'backend' pods on port 8080
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
spec:
  podSelector:
    matchLabels:
      tier: backend
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: frontend
    ports:
    - port: 8080
⚠️ Heads Up

NetworkPolicies only work if your CNI plugin supports them (Calico, Cilium, Weave). The default Flannel CNI does NOT enforce NetworkPolicies.

Workload Types

Deployments are for stateless apps — but Kubernetes has specialized workload controllers for different needs.

📦
Deployment
Stateless apps. Rolling updates, rollback. The default choice for web servers, APIs, and microservices.
statelessrolling updates
🗄️
StatefulSet
Stateful apps that need stable network IDs, ordered scaling, and persistent storage. Databases, message queues.
statefulordered
🌐
DaemonSet
Runs exactly one Pod per Node. Perfect for node-level agents: log collectors, monitoring exporters, CNI plugins.
per-nodeinfra
⚡
Job
Runs Pods to completion. Guarantees N completions. For one-off tasks: DB migrations, batch processing, data imports.
batchfinite
⏰
CronJob
Creates Jobs on a schedule (cron syntax). Nightly reports, cleanup tasks, scheduled backups.
scheduledcron

// StatefulSet Deep Dive

StatefulSets give each Pod a stable identity that persists across rescheduling. Where Deployment Pods are interchangeable, StatefulSet Pods are unique.

Deployment
Pods: web-abc12, web-xyz99 (random)
Any pod can be deleted/replaced first
Shared or no persistent storage
Scale up/down in any order
StatefulSet
Pods: db-0, db-1, db-2 (stable ordinal)
Scales up/down in strict order
Each pod gets its OWN PVC
Stable DNS: db-0.db-svc.ns.svc.cluster.local
yaml · statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres-svc   # headless service required
  replicas: 3
  selector:
    matchLabels: { app: postgres }
  template:
    metadata:
      labels: { app: postgres }
    spec:
      containers:
      - name: postgres
        image: postgres:15
        volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:         # each pod gets its own PVC!
  - metadata:
      name: data
    spec:
      accessModes: [ReadWriteOnce]
      resources:
        requests:
          storage: 20Gi

// DaemonSet

yaml · daemonset — node log collector
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels: { app: fluentd }
  template:
    metadata:
      labels: { app: fluentd }
    spec:
      tolerations:
      - key: node-role.kubernetes.io/control-plane
        effect: NoSchedule    # allow on control plane nodes too
      containers:
      - name: fluentd
        image: fluentd:v1.16
        volumeMounts:
        - name: varlog
          mountPath: /var/log
          readOnly: true
      volumes:
      - name: varlog
        hostPath:
          path: /var/log

// Jobs & CronJobs

yaml · job.yaml + cronjob.yaml
# One-off Job
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  completions: 1            # run to completion once
  parallelism: 1            # one pod at a time
  backoffLimit: 3           # retry up to 3 times on failure
  template:
    spec:
      restartPolicy: Never  # required for Jobs
      containers:
      - name: migrate
        image: my-app:1.0
        command: ["python", "manage.py", "migrate"]
---
# Scheduled CronJob
apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-report
spec:
  schedule: "0 2 * * *"       # every night at 2am UTC
  concurrencyPolicy: Forbid  # don't run if previous still running
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 1
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: report
            image: report-generator:latest

Health Probes

Kubernetes can't know if your app is truly healthy just by checking if the container is running. Probes let you define health checks so K8s can make smarter decisions.

💓
Liveness Probe
Is the app alive? Failing = restart the container. Use for deadlock detection.
restart on fail
🚦
Readiness Probe
Is the app ready to serve traffic? Failing = remove from Service endpoints. No restart.
remove from LB
🚀
Startup Probe
For slow-starting apps. Disables liveness/readiness until startup succeeds. Prevents premature restarts.
slow apps

// Probe Types

All three probe types support the same three check mechanisms:

yaml · all three probe types in action
spec:
  containers:
  - name: app
    image: my-app:1.0

    # Startup: give slow apps up to 3 minutes (30 × 6s) to start
    startupProbe:
      httpGet:
        path: /health/startup
        port: 8080
      failureThreshold: 30
      periodSeconds: 6

    # Liveness: restart if /health returns non-200 3 times in a row
    livenessProbe:
      httpGet:
        path: /health/live
        port: 8080
      initialDelaySeconds: 10  # wait 10s before first check
      periodSeconds: 15
      failureThreshold: 3

    # Readiness: only send traffic when DB connections are ready
    readinessProbe:
      httpGet:
        path: /health/ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 10
      successThreshold: 1   # 1 success to become ready again
      failureThreshold: 2   # 2 failures to become unready

// Other Probe Mechanisms

yaml · tcp and exec probes
# TCP Socket (good for databases that don't have HTTP)
livenessProbe:
  tcpSocket:
    port: 5432          # just checks if port is open
  periodSeconds: 20

# Exec Command (run a command inside the container)
livenessProbe:
  exec:
    command:
    - redis-cli
    - ping             # exit code 0 = healthy
  periodSeconds: 10
🔑 Rule of Thumb

Use readiness + liveness together on all production apps. Add startup probe for anything that takes more than 20 seconds to initialize (Java apps, heavy ML models, etc.).

Resource Management

Without resource limits, a single runaway Pod can starve the entire node. Kubernetes has a rich model for controlling CPU and memory.

// Requests & Limits

Requests — what the container is guaranteed to get. Used by the Scheduler to find a Node with enough capacity.
Limits — the maximum it can use. The container is throttled (CPU) or OOM-killed (memory) if it exceeds this.

yaml · resource requests & limits
spec:
  containers:
  - name: app
    image: my-app:1.0
    resources:
      requests:
        cpu: "250m"      # 250 millicores = 0.25 CPU
        memory: "256Mi"  # mebibytes
      limits:
        cpu: "1000m"     # 1 full CPU core max
        memory: "512Mi"
📐 CPU Units

1000m = 1 CPU core. You can also write "1" or "0.5". Memory uses standard suffixes: Ki, Mi, Gi (base-2) or K, M, G (base-10).

// QoS Classes

Based on your requests/limits settings, K8s assigns a QoS class that determines eviction priority under node pressure.

QoS ClassConditionEviction Priority
Guaranteedrequests == limits for all containersLast to be evicted ✅
Burstablerequests < limits (or only requests set)Evicted under pressure
BestEffortno requests or limits at allFirst to be evicted ❌

// LimitRange

Sets default and max resource values for all containers in a namespace. Prevents Pods without resource specs from consuming unbounded resources.

yaml · limitrange.yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: staging
spec:
  limits:
  - type: Container
    default:          # applied if not specified
      cpu: "500m"
      memory: "256Mi"
    defaultRequest:
      cpu: "100m"
      memory: "128Mi"
    max:              # hard ceiling
      cpu: "2"
      memory: "2Gi"

// ResourceQuota

Caps the total resource consumption for an entire namespace. Ensures one team can't exhaust shared cluster resources.

yaml · resourcequota.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: team-payments
spec:
  hard:
    pods: "20"               # max 20 pods in this namespace
    requests.cpu: "4"        # total CPU requests <= 4 cores
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    persistentvolumeclaims: "5"
    services.loadbalancers: "2"  # controls cloud LB costs

# Check quota usage
kubectl describe resourcequota team-quota -n team-payments

RBAC — Role-Based Access Control

RBAC controls who can do what in your cluster. It answers: "Can this user/service list Pods in the production namespace?"

RBAC Model
Subject
(User / Group / ServiceAccount)
→ bound via →
RoleBinding /
ClusterRoleBinding
→ grants →
Role /
ClusterRole
→ allows →
verbs on resources
(get, list, create...)
ObjectScopeUse Case
RoleNamespaceGrant access within a single namespace
ClusterRoleCluster-wideGrant access across all namespaces or to cluster-level resources (Nodes, PVs)
RoleBindingNamespaceAttach a Role (or ClusterRole) to a subject in a namespace
ClusterRoleBindingCluster-wideAttach a ClusterRole to a subject cluster-wide
ServiceAccountNamespaceIdentity for Pods and in-cluster processes

// Role & RoleBinding

yaml · role + rolebinding
# Role: allow read-only access to pods and logs in 'dev' namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
- apiGroups: [""]                # "" = core API group
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
# RoleBinding: assign pod-reader role to user 'alice'
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: alice-pod-reader
  namespace: dev
subjects:
- kind: User
  name: alice              # must match the user's CN in their kubeconfig cert
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

// ServiceAccount

ServiceAccounts are identities for Pods. When your app needs to call the Kubernetes API (e.g., a CI/CD tool, an operator, a monitoring agent), it uses a ServiceAccount.

yaml + bash · serviceaccount with permissions
# Create a ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: default
---
# Bind a ClusterRole to it
kind: RoleBinding
metadata:
  name: my-app-binding
subjects:
- kind: ServiceAccount
  name: my-app-sa
  namespace: default
roleRef:
  kind: ClusterRole
  name: view              # built-in read-only ClusterRole
  apiGroup: rbac.authorization.k8s.io
---
# Use the ServiceAccount in a Pod
spec:
  serviceAccountName: my-app-sa
  containers: [...]
bash · test RBAC permissions
# Can I do this? (auth can-i)
kubectl auth can-i list pods --namespace production
yes

kubectl auth can-i delete nodes
no

# Can alice do this?
kubectl auth can-i list pods --as alice -n dev
yes

# List all permissions for current user
kubectl auth can-i --list -n dev
🔐 Security Principle

Always follow the principle of least privilege. Start with the minimal permissions needed and add more only when required. Avoid ClusterAdmin for regular users or application ServiceAccounts.

Let's wire everything from this stage into a realistic multi-tier application setup on Minikube.

📋 Prerequisites

Minikube running with ingress addon enabled: minikube addons enable ingress

1

Deploy a StatefulSet (PostgreSQL)

Use a StatefulSet for a database that needs stable storage and identity.

bash
# Create the namespace
kubectl create namespace lab

# Apply StatefulSet + headless service
kubectl apply -f https://k8s.io/examples/application/guestbook/redis-leader-deployment.yaml -n lab

kubectl apply -f pvc.yaml -n lab

NAME     STATUS   VOLUME       CAPACITY   ACCESS MODES
my-pvc   Bound    pvc-abc123   5Gi        RWO
2

Add Health Probes to a Deployment

Add liveness and readiness probes to see how K8s responds to failure.

bash
# Deploy nginx with health probes
kubectl create deployment webapp --image=nginx -n lab

# Edit and add probes
kubectl edit deployment webapp -n lab

# Watch pod restarts after probe failure
kubectl get pods -n lab -w

# Describe to see probe events
kubectl describe pod <pod-name> -n lab
3

Set Resource Limits + LimitRange

Apply a LimitRange to the namespace and observe quota enforcement.

bash
kubectl apply -f limitrange.yaml -n lab
kubectl describe limitrange -n lab

# Try to deploy a container exceeding the limit — should be rejected

4

Create an Ingress

Expose the webapp via an Ingress rule with a hostname.

bash
# Expose service first
kubectl expose deployment webapp --port=80 -n lab

# Apply ingress

# Add to /etc/hosts
echo "$(minikube ip) myapp.local" | sudo tee -a /etc/hosts

# Test
curl http://myapp.local
<!DOCTYPE html>...  ← nginx welcome page via Ingress!
5

Create an RBAC Role

Create a read-only ServiceAccount and test its permissions.

bash
# Create ServiceAccount
kubectl create serviceaccount viewer -n lab

yes

kubectl auth can-i delete deployments --as=system:serviceaccount:lab:viewer -n lab
no
6

Run a CronJob

Schedule a job that runs every minute and verify it executes.

bash

# Watch jobs being created

# See the output
kubectl logs job/hello-<hash> -n lab
