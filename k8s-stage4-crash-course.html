Helm — The Kubernetes Package Manager

Deploying a real application to Kubernetes means managing dozens of YAML files — Deployments, Services, ConfigMaps, Ingress, RBAC, and more. Helm bundles all of this into a reusable, versioned package called a Chart.

Helm Architecture
Chart
templates/ + values.yaml
+
values.yaml
your overrides
→ helm install →
Release
running in cluster
Chart Repository
Artifact Hub, ECR, OCI
→ helm repo add / pull →
Local Chart Cache
→ helm upgrade →
Updated Release

// Core Concepts

Chart — a Helm package. Contains Kubernetes YAML templates + a values.yaml for defaults.
Release — an instance of a Chart installed in a cluster. You can install the same Chart multiple times with different names.
Repository — a collection of Charts, hosted remotely (like npm registry or Docker Hub).
Values — configuration that gets injected into templates at install/upgrade time.

// Writing Helm Templates

Templates use Go templating syntax. Values from values.yaml are injected via {{ .Values.key }}.

yaml · templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "myapp.fullname" . }}
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      {{- include "myapp.selectorLabels" . | nindent 6 }}
  template:
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"
        ports:
        - containerPort: {{ .Values.service.port }}
        {{- if .Values.resources }}   ← conditional block
        resources:
          {{- toYaml .Values.resources | nindent 10 }}
        {{- end }}
yaml · values.yaml — defaults
replicaCount: 2

image:
  repository: nginx
  tag: ""          # defaults to chart's appVersion
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 80

ingress:
  enabled: false   # feature flags — enable per environment
  className: nginx
  host: app.example.com

resources:
  requests: { cpu: 100m, memory: 128Mi }
  limits:   { cpu: 500m, memory: 256Mi }
bash · overriding values per environment
# production overrides (values-prod.yaml)
# replicaCount: 5, ingress.enabled: true, etc.

helm upgrade --install myapp ./myapp \
  -f values.yaml \
  -f values-prod.yaml \          # layered — prod overrides base
  --set image.tag=v2.3.1 \      # override single value inline
  --namespace production \
  --create-namespace \
  --wait \                       # block until all pods are ready
  --timeout 5m

CI/CD Integration

CI/CD pipelines automate building, testing, and deploying your application. In a Kubernetes world, the pipeline builds a container image, pushes it to a registry, and then updates the cluster.

→
Push to Registry

→

// GitHub Actions Pipeline

GitHub Actions is the most common CI/CD choice for Kubernetes projects. Here's a production-grade pipeline that builds, tests, and deploys on every push to main.

yaml · .github/workflows/deploy.yaml
name: Build and Deploy

on:
  push:
    branches: [main]

env:
  IMAGE: ghcr.io/${{ github.repository }}/app
  TAG: ${{ github.sha }}

jobs:
  # ── Job 1: Build & push Docker image ──────────────────
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4

    - name: Log in to GHCR
      uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}

    - name: Build and push
      uses: docker/build-push-action@v5
      with:
        push: true
        tags: ${{ env.IMAGE }}:${{ env.TAG }},${{ env.IMAGE }}:latest
        cache-from: type=gha     # GitHub Actions layer cache
        cache-to: type=gha,mode=max

  # ── Job 2: Deploy to staging first ───────────────────
  deploy-staging:
    needs: build
    runs-on: ubuntu-latest
    environment: staging
    steps:
    - uses: actions/checkout@v4

    - name: Configure kubectl
      uses: azure/k8s-set-context@v3
      with:
        kubeconfig: ${{ secrets.KUBECONFIG_STAGING }}

    - name: Deploy with Helm
      run: |
        helm upgrade --install myapp ./helm/myapp \
          --namespace staging \
          --set image.tag=${{ env.TAG }} \
          --wait --timeout 5m

  # ── Job 3: Production (requires approval) ─────────────
  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production   # ← GitHub env with required reviewers
    steps:
    - name: Deploy with Helm
      run: |
        helm upgrade --install myapp ./helm/myapp \
          --namespace production \
          --set image.tag=${{ env.TAG }} \
          -f helm/values-prod.yaml \
          --atomic                # rollback automatically on failure
✅ Best Practices

Always tag images with the git commit SHA (not latest) — this makes every deployment traceable and rollback deterministic. Use GitHub Environments to require human approval before production deploys.

GitOps with ArgoCD

Traditional CI/CD pushes changes to the cluster. GitOps flips this: the cluster pulls its desired state from a Git repository and continuously reconciles to match it. Git becomes the single source of truth.

Traditional CI/CD (Push)
Pipeline has cluster credentials
Pipeline runs kubectl/helm commands
Cluster state can drift without detection
Hard to audit — who deployed what?
Rollback = re-run old pipeline
GitOps (Pull — ArgoCD)
Cluster pulls from Git — no external credentials needed
ArgoCD agent runs inside the cluster
Drift detected and auto-corrected in real time
Full audit trail — every change is a git commit
Rollback = git revert (instant)

// ArgoCD Architecture

GitOps Reconciliation Loop
Dev pushes code
git push
→
CI updates image tag
in Git repo manifests
→
ArgoCD detects diff
desired ≠ actual
→
ArgoCD syncs
applies to cluster
ArgoCD polls every 3 minutes or via webhook. Sync can be automatic or manual with approval.

// Defining an ArgoCD Application

yaml · argocd-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-production-app
  namespace: argocd
spec:
  project: default

  source:
    repoURL: https://github.com/my-org/k8s-configs
    targetRevision: HEAD      # or a specific branch/tag/commit
    path: apps/my-app/production   # path within the repo
    helm:                           # if using Helm charts
      valueFiles:
        - values.yaml
        - values-prod.yaml

  destination:
    server: https://kubernetes.default.svc  # in-cluster
    namespace: production

  syncPolicy:
    automated:
      prune: true      # delete resources removed from Git
      selfHeal: true   # auto-correct drift
    syncOptions:
      - CreateNamespace=true
      - ServerSideApply=true

📁 Repo Structure Tip

Keep a separate config repo from your app repo. The config repo holds all Kubernetes manifests and Helm values. The CI pipeline updates image tags in the config repo. ArgoCD watches the config repo. This separation prevents a single repo from mixing code and infrastructure concerns.

// Prometheus Query Language (PromQL)

PromQL is how you query Prometheus data. You'll use it to write Grafana panels and alert rules.

promql · essential queries
# CPU usage per pod (as percentage of limit)
rate(container_cpu_usage_seconds_total{container!=""}[5m])
  / on(pod) kube_pod_container_resource_limits{resource="cpu"} * 100

# Memory usage per namespace
sum by (namespace) (container_memory_working_set_bytes{container!=""})

# Pod restart count in the last hour
increase(kube_pod_container_status_restarts_total[1h]) > 0

# HTTP request rate per service (5xx errors)
sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)

# Node disk usage %
(node_filesystem_size_bytes - node_filesystem_free_bytes)
  / node_filesystem_size_bytes * 100

# Pods not running
kube_pod_status_phase{phase!="Running",phase!="Succeeded"} == 1

// PrometheusRule — Alerting

yaml · prometheusrule.yaml — custom alert
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: my-app-alerts
  namespace: monitoring
  labels:
    release: kube-prom-stack   # must match for auto-discovery
spec:
  groups:
  - name: my-app.rules
    rules:
    - alert: HighErrorRate
      expr: |
        sum(rate(http_requests_total{status=~"5..",job="my-app"}[5m]))
          / sum(rate(http_requests_total{job="my-app"}[5m])) > 0.01
      for: 2m                 # must be true for 2 mins before firing
      labels:
        severity: critical
      annotations:
        summary: "Error rate {{ $value | humanizePercentage }} in my-app"
        runbook_url: https://wiki.company.com/runbooks/high-error-rate

    - alert: PodCrashLooping
      expr: rate(kube_pod_container_status_restarts_total[15m]) * 60 > 1
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "Pod {{ $labels.pod }} is crash-looping"

Auto-Scaling: HPA & VPA

Kubernetes can automatically scale your workloads based on real-time metrics — no more over-provisioning for peak load or under-provisioning for off-peak.

↔️
HPA — Horizontal Pod Autoscaler
Scales the number of Pods based on CPU, memory, or custom metrics. The most common autoscaler in production.
scale out/inpod count
↕️
VPA — Vertical Pod Autoscaler
Adjusts CPU/memory requests and limits of individual containers. Requires pod restart to apply. Good for right-sizing.
resize resourcesright-sizing
🖥️
Cluster Autoscaler
Adds/removes Nodes from the cluster when Pods can't be scheduled due to resource constraints. Works with cloud provider node groups.
node scalingcloud

// HPA — Horizontal Pod Autoscaler

bash + yaml · HPA setup
# Prerequisites: metrics-server must be installed
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

# Verify it's working
kubectl top pods
kubectl top nodes

# Quick HPA from command line
kubectl autoscale deployment my-app \
  --min=2 --max=10 --cpu-percent=60
yaml · hpa.yaml — full spec with multi-metric
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 20

  metrics:
  # Scale on CPU utilization
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60   # target 60% CPU across pods

  # Scale on memory too
  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: 200Mi

  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300   # wait 5m before scaling down
      policies:
      - type: Pods
        value: 2
        periodSeconds: 60              # remove at most 2 pods per minute
    scaleUp:
      stabilizationWindowSeconds: 0     # scale up immediately
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15             # can double pods every 15s
bash · watch HPA in action
# Watch HPA status
kubectl get hpa my-app-hpa -w
NAME         REFERENCE           TARGETS    MINPODS  MAXPODS  REPLICAS
my-app-hpa  Deployment/my-app  23%/60%    2        20       2
my-app-hpa  Deployment/my-app  78%/60%    2        20       4        ← scaling up!
my-app-hpa  Deployment/my-app  61%/60%    2        20       4

# Simulate load to trigger scaling
kubectl run load-gen --image=busybox --rm -it -- \
  /bin/sh -c "while true; do wget -q -O- http://my-app; done"

// VPA — Vertical Pod Autoscaler

yaml · vpa.yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  updatePolicy:
    updateMode: Auto     # Off | Initial | Recreate | Auto
  resourcePolicy:
    containerPolicies:
    - containerName: '*'
      minAllowed: { cpu: 50m, memory: 64Mi }
      maxAllowed: { cpu: 2,   memory: 2Gi  }

# Check VPA recommendations (great in 'Off' mode for right-sizing)
kubectl describe vpa my-app-vpa
  Recommendation:
    Container Recommendations:
      Container Name: my-app
        Lower Bound:   cpu: 50m,  memory: 100Mi
        Target:        cpu: 120m, memory: 230Mi   ← what VPA suggests
        Upper Bound:   cpu: 500m, memory: 1Gi
⚠️ HPA + VPA Conflict

Don't use HPA on CPU/memory and VPA on the same deployment simultaneously — they will fight. Use HPA for pod count scaling and VPA in Off mode for resource recommendations only, then apply manually.

Multi-Environment Management

Every real organization needs dev, staging, and production environments. Kubernetes gives you multiple strategies — each with different trade-offs.

Namespaces (single cluster)
Simple — one cluster, multiple namespaces
Cheaper — shared control plane & nodes
Logical isolation with RBAC + NetworkPolicy
Risk: noisy neighbor problems between envs
Good for dev/staging on the same cluster
Separate Clusters
True isolation — blast radius is contained
Production has no shared fate with staging
Different K8s versions per env possible
Higher cost & more operational overhead
Industry standard for production

// Namespace Strategy

bash · namespace-per-environment pattern
# Create namespaces
kubectl create namespace development
kubectl create namespace staging
kubectl create namespace production

# Label them for policy enforcement
kubectl label namespace production env=production tier=prod
kubectl label namespace staging    env=staging   tier=nonprod
kubectl label namespace development env=dev      tier=nonprod

# Apply different resource quotas per env
kubectl apply -f quota-dev.yaml   -n development
kubectl apply -f quota-staging.yaml -n staging
kubectl apply -f quota-prod.yaml  -n production

# Deploy to each env with different values
helm upgrade --install myapp ./chart -n development -f values-dev.yaml
helm upgrade --install myapp ./chart -n staging     -f values-staging.yaml
helm upgrade --install myapp ./chart -n production  -f values-prod.yaml

// Managing Multiple Cluster Contexts

When working across multiple clusters (dev cluster, staging cluster, prod cluster), kubectl config manages your credentials and context switching.

// Kustomize — Environment Overlays

Kustomize (built into kubectl) lets you maintain a base configuration and apply environment-specific overlays — without templating syntax.

bash · kustomize directory structure
k8s/
├── base/
│   ├── kustomization.yaml    ← lists all base resources
│   ├── deployment.yaml
│   └── service.yaml
└── overlays/
    ├── staging/
    │   ├── kustomization.yaml  ← patches on top of base
    │   └── patch-replicas.yaml  (replicas: 1)
    └── production/
        ├── kustomization.yaml
        └── patch-replicas.yaml  (replicas: 5)
yaml · overlays/production/kustomization.yaml
resources:
  - ../../base

images:
  - name: my-app
    newTag: v2.3.1   # override image tag

patches:
  - patch: |-
      - op: replace
        path: /spec/replicas
        value: 5
    target:
      kind: Deployment
      name: my-app

# Apply directly
kubectl apply -k overlays/production
kubectl diff -k overlays/production    # preview changes

// Getting Started with EKS

bash · create EKS cluster with eksctl
# Install eksctl
brew install eksctl

# Create cluster (takes ~15 minutes)
eksctl create cluster \
  --name my-cluster \
  --region us-east-1 \
  --node-type t3.medium \
  --nodes-min 2 \
  --nodes-max 10 \
  --managed              # AWS-managed node group

# Update kubeconfig to use new cluster
aws eks update-kubeconfig --name my-cluster --region us-east-1

# Verify
kubectl get nodes
NAME                          STATUS   ROLES    AGE   VERSION
ip-10-0-1-45.ec2.internal    Ready       5m    v1.29.3

# Enable Cluster Autoscaler
eksctl create iamserviceaccount \
  --name cluster-autoscaler \
  --namespace kube-system \
  --cluster my-cluster \
  --attach-policy-arn arn:aws:iam::aws:policy/AutoScalingFullAccess \
  --approve

// Getting Started with AKS

bash · create AKS cluster
# Create resource group
az group create --name myRG --location eastus

# Create AKS cluster
az aks create \
  --resource-group myRG \
  --name my-cluster \
  --node-count 3 \
  --node-vm-size Standard_D4s_v3 \
  --enable-cluster-autoscaler \
  --min-count 2 --max-count 10 \
  --enable-workload-identity \
  --enable-oidc-issuer

# Get credentials

💰 Cost Tip

For learning, use Spot/Preemptible/Spot instances for non-production node groups — up to 90% cheaper. EKS calls them Spot, GKE calls them Spot VMs, AKS calls them Spot node pools. Just ensure your workloads tolerate eviction.

Create and Deploy a Helm Chart

Scaffold a chart, customise values, and deploy to Minikube.

bash
# Scaffold chart
helm create my-webserver
cd my-webserver

# Lint and render templates before installing
helm lint .
helm template my-release . --set replicaCount=3

# Install to Minikube
helm install my-release . \
  --set replicaCount=3 \
  --set service.type=NodePort

# Verify
helm status my-release
kubectl get all

# Upgrade (bump replicas)
helm upgrade my-release . --set replicaCount=5
helm history my-release

# Rollback
helm rollback my-release 1
2

Install ArgoCD and Connect a Repo

Install ArgoCD and deploy an application declaratively via Git.

bash
# Install ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
kubectl wait pod -l app.kubernetes.io/name=argocd-server \
  -n argocd --for=condition=Ready --timeout=120s

# Access UI
kubectl port-forward svc/argocd-server -n argocd 8080:443 &
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

# Deploy a sample app from public repo
kubectl apply -f - <<EOF
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: guestbook
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/argoproj/argocd-example-apps
    path: guestbook
    targetRevision: HEAD
  destination:
    server: https://kubernetes.default.svc
    namespace: guestbook
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions: [CreateNamespace=true]
EOF

kubectl get app guestbook -n argocd

Deploy Prometheus + Grafana

Install the monitoring stack and explore a dashboard.

bash
helm repo add prometheus-community \
  https://prometheus-community.github.io/helm-charts
helm repo update

helm upgrade --install monitoring \
  prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace \
  --set grafana.adminPassword=admin123 \
  --wait

# Open Grafana
kubectl port-forward -n monitoring \
  svc/monitoring-grafana 3000:80 &

# Browse to http://localhost:3000 (admin/admin123)
# Explore: Dashboards → Kubernetes → Compute Resources → Cluster

# Query Prometheus directly
kubectl port-forward -n monitoring \
  svc/monitoring-kube-prometheus-prometheus 9090:9090 &
# Browse to http://localhost:9090 and try PromQL queries

Configure HPA and Trigger It

Set up autoscaling and generate load to watch it in action.

bash
# Install metrics-server
kubectl apply -f \
  https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

# Deploy a CPU-intensive app
kubectl create deployment php-apache \
  --image=registry.k8s.io/hpa-example
kubectl set resources deployment php-apache \
  --requests=cpu=200m
kubectl expose deployment php-apache --port=80

# Create HPA
kubectl autoscale deployment php-apache \
  --cpu-percent=50 --min=1 --max=10

# Terminal 1: watch HPA
kubectl get hpa -w

# Terminal 2: generate load
kubectl run loader --image=busybox --rm -it \
  -- /bin/sh -c "while true; do wget -q -O- http://php-apache; done"

Multi-Environment with Kustomize

Create base + overlay directories and apply environment-specific configs.

bash
mkdir -p k8s/{base,overlays/{dev,prod}}

# Base deployment
cat > k8s/base/kustomization.yaml <<EOF
resources:
  - deployment.yaml
  - service.yaml
EOF

# Dev overlay: 1 replica
cat > k8s/overlays/dev/kustomization.yaml <<EOF
resources: ["../../base"]
patches:
  - patch: |-
      - op: replace
        path: /spec/replicas
        value: 1
    target: {kind: Deployment, name: my-app}
EOF

# Preview diff between dev and prod
kubectl diff -k k8s/overlays/dev
kubectl apply -k k8s/overlays/dev -n development
