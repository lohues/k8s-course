CRDs & Operators

Kubernetes is extensible. Custom Resource Definitions (CRDs) let you teach Kubernetes about your own resource types. An Operator is a controller that uses CRDs to automate the lifecycle of complex stateful applications — encoding operational knowledge into software.

// Defining a CRD

yaml · custom resource definition
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: databases.mycompany.io    # plural.group
spec:
  group: mycompany.io
  names:
    kind: Database
    plural: databases
    singular: database
    shortNames: [db]
  scope: Namespaced
  versions:
  - name: v1alpha1
    served: true
    storage: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            required: [engine, version, replicas]
            properties:
              engine:
                type: string
                enum: [postgres, mysql, mariadb]
              version:
                type: string
              replicas:
                type: integer
                minimum: 1
                maximum: 5
              storage:
                type: string
                default: "10Gi"
    additionalPrinterColumns:
    - name: Engine
      type: string
      jsonPath: .spec.engine
    - name: Ready
      type: string
      jsonPath: .status.ready

// Writing an Operator in Go (with controller-runtime)

The Operator SDK and controller-runtime are the standard frameworks. An Operator is a Go program running in the cluster that watches for your CRs and reconciles the actual state to match the desired state.

go · database_controller.go — reconcile loop skeleton
package controllers

import (
    "context"
    appsv1 "k8s.io/api/apps/v1"
    myv1  "github.com/myorg/db-operator/api/v1alpha1"
    ctrl  "sigs.k8s.io/controller-runtime"
)

type DatabaseReconciler struct {
    client.Client
    Scheme *runtime.Scheme
}

// Reconcile is called whenever a Database CR is created/updated/deleted
func (r *DatabaseReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    // 1. Fetch the Database CR
    db := &myv1.Database{}
    if err := r.Get(ctx, req.NamespacedName, db); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }

    // 2. Define desired StatefulSet
    sts := r.buildStatefulSet(db)

    // 3. Create or Update (server-side apply pattern)
    if err := ctrl.SetControllerReference(db, sts, r.Scheme); err != nil {
        return ctrl.Result{}, err
    }
    if err := r.createOrUpdate(ctx, sts); err != nil {
        return ctrl.Result{}, err
    }

    // 4. Update status subresource
    db.Status.Ready = "true"
    db.Status.Replicas = *sts.Spec.Replicas
    r.Status().Update(ctx, db)

    // 5. Requeue after 30s to drift-check
    return ctrl.Result{RequeueAfter: 30 * time.Second}, nil
}

The Operator Framework defines 5 capability levels: Level 1 (Basic Install) → Level 2 (Seamless Upgrades) → Level 3 (Full Lifecycle) → Level 4 (Deep Insights) → Level 5 (Auto Pilot). Start at Level 1 and add capabilities as your operator matures.

Service Mesh with Istio

At scale, microservices need more than Kubernetes provides: mutual TLS between every service, fine-grained traffic control, circuit breaking, distributed tracing, and deep observability — without changing application code. That's what a service mesh delivers.

Without Service Mesh
mTLS requires app-level implementation
Traffic splitting needs multiple Deployments
Retry/timeout logic in every service
No automatic distributed tracing
Observability gaps between services
With Istio (Sidecar Proxy)
Automatic mTLS for all Pod-to-Pod traffic
Traffic splitting via VirtualService weights
Retry/timeout declared in YAML, not code
Automatic Jaeger/Zipkin distributed traces
L7 metrics: latency, error rate, throughput per route

// Traffic Management

yaml · canary deploy — 90/10 traffic split
# VirtualService controls routing
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata: { name: my-app }
spec:
  hosts: [my-app]
  http:
  - route:
    - destination: { host: my-app, subset: stable }
      weight: 90
    - destination: { host: my-app, subset: canary }
      weight: 10
    retries:
      attempts: 3
      perTryTimeout: 5s
    timeout: 10s

---
# DestinationRule defines subsets (stable vs canary)
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata: { name: my-app }
spec:
  host: my-app
  trafficPolicy:
    tls: { mode: ISTIO_MUTUAL }     # enforce mTLS
    connectionPool:
      tcp: { maxConnections: 100 }
    outlierDetection:                  # circuit breaker
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 60s
  subsets:
  - name: stable
    labels: { version: v1 }
  - name: canary
    labels: { version: v2 }

Advanced Security Hardening

Production clusters handle sensitive data and run business-critical workloads. Security must be layered: admission control, runtime detection, supply chain integrity, and secret management.

// OPA Gatekeeper — Policy as Code

Open Policy Agent (OPA) Gatekeeper enforces custom policies at admission time — before any resource reaches etcd. Write policies in Rego language, define them as ConstraintTemplates.

bash + yaml · install gatekeeper and create a policy
# Install Gatekeeper
helm repo add gatekeeper https://open-policy-agent.github.io/gatekeeper/charts
helm install gatekeeper gatekeeper/gatekeeper --namespace gatekeeper-system \
  --create-namespace

---
# ConstraintTemplate: defines the policy logic in Rego
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata: { name: requiredlabels }
spec:
  crd:
    spec:
      names: { kind: RequiredLabels }
      validation:
        openAPIV3Schema:
          type: object
          properties:
            labels: { type: array, items: { type: string } }
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package requiredlabels
      violation[{"msg": msg}] {
          required := input.parameters.labels[_]
          not input.review.object.metadata.labels[required]
          msg := sprintf("Missing required label: %v", [required])
      }

---
# Constraint: enforce the policy on Deployments
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: RequiredLabels
metadata: { name: deployment-must-have-owner }
spec:
  match:
    kinds: [{ apiGroups: [apps], kinds: [Deployment] }]
  parameters:
    labels: [owner, team, cost-center]

// Pod Security Standards

Kubernetes built-in Pod Security Admission (replaced PodSecurityPolicy in 1.25) enforces security profiles at the namespace level via labels.

bash · pod security admission labels
# Three levels: privileged | baseline | restricted
# Three modes:  enforce | audit | warn

# Label a namespace to enforce restricted policy
kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/enforce-version=latest \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/warn=restricted

# A "restricted" pod must meet these requirements:
- Run as non-root user
- Drop ALL capabilities
- No privilege escalation
- Read-only root filesystem (recommended)
- Use seccompProfile: RuntimeDefault or Localhost
yaml · restricted-compliant pod spec
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 10001
    fsGroup: 10001
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: myapp:v1
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: [ALL]
    volumeMounts:
    - name: tmp
      mountPath: /tmp             # writeable tmpfs for /tmp
  volumes:
  - name: tmp
    emptyDir: { medium: Memory }

// Falco — Runtime Threat Detection

Falco monitors kernel system calls in real-time and alerts on suspicious behavior — like a shell being spawned inside a container, a sensitive file being read, or privilege escalation.

bash + yaml · install falco and write a custom rule
helm repo add falcosecurity https://falcosecurity.github.io/charts
helm install falco falcosecurity/falco \
  --namespace falco --create-namespace \
  --set driver.kind=ebpf \           # use eBPF for modern kernels
  --set falcosidekick.enabled=true \ # forward alerts to Slack/PagerDuty
  --set falcosidekick.config.slack.webhookurl=https://hooks.slack.com/...

# Custom Falco rule — alert if anyone exec's into a prod pod
- rule: Exec into Production Container
  desc: Alert when kubectl exec is used on a production pod
  condition: spawned_process and container and k8s.ns.name = "production"
  output: |
    Shell opened in production (user=%user.name pod=%k8s.pod.name
    cmd=%proc.cmdline ns=%k8s.ns.name)
  priority: WARNING
  tags: [production, shell, compliance]

etcd Operations

etcd is Kubernetes's brain — every cluster resource, every Secret, every ConfigMap lives here. Losing etcd without a backup means losing your entire cluster state. Mastering etcd operations is critical for CKA and production reliability.

etcd Raft Consensus in a 3-Node Cluster
etcd Leader
receives all writes
replicates to followers
← replicate →
etcd Follower 1
votes, serves reads
etcd Follower 2
votes, serves reads
Quorum = (n/2)+1 nodes must agree. 3-node cluster can lose 1 node. 5-node can lose 2. Always use odd numbers.

══════════════════════════════════════════
  RESTORE — disaster recovery
══════════════════════════════════════════
# 1. Stop API server (move static pod manifest out)
mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/

# 2. Restore snapshot to a new data directory
etcdctl snapshot restore /backup/etcd-snapshot-20240225-0300.db \
  --data-dir /var/lib/etcd-restored \
  --name cp-node-1 \
  --initial-cluster cp-node-1=https://10.0.0.1:2380 \
  --initial-cluster-token etcd-cluster-restored \
  --initial-advertise-peer-urls https://10.0.0.1:2380

# 3. Update etcd static pod to use new data dir
sed -i 's|/var/lib/etcd|/var/lib/etcd-restored|g' \
  /etc/kubernetes/manifests/etcd.yaml

# 4. Restore API server
mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# 5. Verify cluster is back
kubectl get nodes
kubectl get pods -A
⚠️ Backup Frequency

Run etcd snapshots at minimum every hour for production clusters. Use a CronJob that uploads snapshots to object storage (S3, GCS) with lifecycle rules to retain 7 days. Test your restore procedure quarterly — a backup you've never tested is not a backup.

      project: default

        targetRevision: HEAD
      destination:
        server: '{{server}}'         # cluster API server URL
        namespace: production
      syncPolicy:
        automated: { prune: true, selfHeal: true }

Advanced Debugging Techniques

At the advanced level, you need tools and techniques that go beyond kubectl logs. When things break in production, you need to move fast and dig deep.

// Ephemeral Debug Containers

Debug containers let you attach a fully-tooled shell into a running Pod without modifying its spec — even if the Pod's image has no shell at all.

bash · ephemeral debug containers
# Attach a debug container to a running pod
kubectl debug -it my-pod \
  --image=nicolaka/netshoot \     # has tcpdump, curl, dig, ss, nmap etc.
  --target=my-container           # share PID namespace with this container

# Clone pod to a node with a debug container (doesn't affect original)
kubectl debug -it my-pod \
  --copy-to=debug-pod \
  --image=busybox \
  --share-processes

# Debug a node directly (runs privileged pod on the node)
kubectl debug node/worker-node-1 -it --image=ubuntu
# Inside: chroot /host to access the node's full filesystem

// Deep Network Debugging

bash · network packet capture in a pod
# Capture traffic on a pod's network interface
kubectl debug -it my-pod --image=nicolaka/netshoot --target=app
# Inside the debug container:
tcpdump -i eth0 -nn 'port 5432'       # watch Postgres traffic

// API Server Audit Logs

Audit logs record every request to the API server — who did what, when, and what the response was. Essential for security investigations and compliance.

yaml · audit-policy.yaml
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
# Log all Secrets access at RequestResponse level
- level: RequestResponse
  resources:
  - group: ""
    resources: [secrets]

# Log Pod exec/attach at Metadata level
- level: Metadata
  resources:
  - group: ""
    resources: [pods/exec, pods/attach, pods/portforward]

# Skip noisy read-only operations
- level: None
  verbs: [get, list, watch]
  resources:
  - group: ""
    resources: [endpoints, services, configmaps]

# Catch-all — log everything else at Metadata
- level: Metadata

# Add to kube-apiserver:
# --audit-policy-file=/etc/kubernetes/audit-policy.yaml
